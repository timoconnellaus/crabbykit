## Why

CLAW agents today are statically defined at worker compile time: the system prompt, model, tools, and capabilities are all baked in by `defineAgent()`, and changing any of them requires redeploying the worker. We want a way for agents to override their own brain at runtime — load a new system prompt, swap a model, add a tool, change a capability set — without redeploying. The use cases are (a) agents that author and iterate on their own behavior at runtime, (b) per-deployment agent variants without per-variant workers, and (c) self-extending agents whose LLM authors new behavior.

A previous attempt (`add-agent-bundles`) tried to solve this by introducing a parallel "loader-backed agent" SDK alongside `defineAgent`, with its own DO class, runtime, and adapter interfaces. Three rounds of adversarial review surfaced architectural blockers (sync `SessionStore` interface broke when async-ified across the loader boundary, isolate pinning contradicted content-addressed cache keys, per-turn token construction conflicted with per-session warm state) plus a UX problem (every consumer paid bundle infrastructure overhead even if they didn't want it). The fundamental error was treating bundles as a peer to static agents rather than as **a brain override on the same agent**.

This change takes the simpler approach. The agent's identity, DO, session state, transport, and lifecycle are unchanged — those remain the static `AgentRuntime` we already have. What gets added is one optional config field on `defineAgent` that lets the agent's *inference brain* (system prompt, model, tools, capabilities, hook chain, LLM loop) be overridden at runtime by a bundle loaded via Cloudflare Worker Loader. If a bundle is registered for the agent, the bundle's brain runs the turn; if not, the agent's static brain runs. The static brain is always the safety net — it's the agent's compile-time identity and its de facto recovery path. Self-editing is safe by construction because removing the bundle reverts to the static brain on the next turn.

## What Changes

- **`defineAgent` gains one optional `bundle` config field.** When omitted (today's case), the agent is purely static — exact same code path, exact same wrangler config, exact same dependencies. When provided, the agent gains the ability to load a runtime-active bundle from a registry on each turn and dispatch the turn's inference loop into that bundle instead of the static one.
- **One DO per agent, unchanged.** The agent's wrangler-registered DO class is the same `defineAgent`-produced class as today. The DO owns sessions, alarms, transport, hibernation, in-flight turn tracking, WebSocket connections — everything it owns today. Adding bundle support to an agent does not change its DO class, its DO ID, its session schema, or any client-facing surface.
- **Bundle = an internal implementation strategy of the same agent**, not a separate agent. From outside (clients, sessions, observers), an agent with bundle support active behaves identically to one without — it's still MyAgent, still has the same sessions, still streams events the same way. The bundle is a backend swap of the inference loop, invisible to anything that doesn't deliberately look at version metadata.
- **Static brain is the always-available fallback.** The fields a bundle would override (`model`, `prompt`, `tools`, `capabilities`) are still required on `defineAgent` — they define the agent's static brain, which runs whenever no bundle is registered and serves as automatic recovery if a bundle becomes inactive. There is no "factory-reset bundle" because the static config is the reset destination.
- **Bundle authoring API: `defineBundleAgent`** (in a new package `@crabbykit/agent-bundle`). Same declarative shape as `defineAgent`'s brain fields, minus DO-specific surface. Authored in TypeScript, compiled with `bun build`, loaded via Worker Loader. `BundleEnv` type-restricted: only `Service<T>` bindings and serializable values; native bindings (`Ai`, `R2Bucket`, `WorkerLoader`, etc.) and provider API keys are forbidden by type.
- **Bundle runtime is a separate, smaller, async-by-default runtime.** Lives in `@crabbykit/agent-bundle`. Runs inside a Worker Loader isolate. Uses an async `SessionStoreClient` that RPCs back to the host DO via a `SpineService` WorkerEntrypoint. The static `AgentRuntime` and `SessionStore` are unchanged — they stay sync. The async/sync impedance is resolved at the service binding boundary on the host side, where `SpineService.appendEntry` (called via JSRPC from the bundle) internally invokes the DO's sync `sessionStore.appendEntry` and returns. **No public API on the static side changes.**
- **Per-turn capability tokens for authorization.** Every RPC method on `SpineService`, `LlmService`, and capability service classes takes a sealed HMAC token as its first argument. The host DO mints the token per turn binding `{agentId, sessionId, expiresAt, nonce}`; services verify it and derive identity from the verified payload. Method signatures do not accept `sessionId` as an argument — identity is structural, not advisory. HKDF-derived per-service verify-only subkeys, so a compromised single service cannot mint tokens.
- **Host-side `LlmService`** in `@crabbykit/agent-bundle`. Multi-provider proxy (OpenRouter, Anthropic, OpenAI, Workers AI). Holds provider API keys in its own env. Bundles declare `model: { provider, modelId }` with no apiKey; the runtime routes through `env.LLM_SERVICE` via service binding with the verified capability token. Static agents continue to use direct provider credentials as today and do not interact with `LlmService`.
- **Capability service pattern** for capabilities that hold secrets. Three subpath exports: `index` (legacy static-agent factory, unchanged), `service` (host-side `WorkerEntrypoint` holding credentials), `client` (bundle-side factory taking a `Service<T>`). Pilot: `packages/tavily-web-search`. Cost emission moves into the service, attributed via the verified token, so bundles cannot suppress or fabricate cost events. Capability packages without secrets get only a `bundle` subpath alongside their `index`.
- **Bundle registry** in `packages/bundle-registry`. D1-backed version store with content-addressed bundle IDs (SHA-256 of compiled artifact bytes), per-agent active-version pointer, append-only deployment audit log. Compiled bytes in KV keyed by version ID. Self-seeding migration following the `skill-registry` pattern. **Deploy uses KV readback verification** with exponential backoff polling because Cloudflare KV is not write-your-own-reads consistent even within a single DO.
- **Bundle workshop capability** in `packages/bundle-workshop`. Tools: `bundle_init`, `bundle_build`, `bundle_test`, `bundle_deploy`, `bundle_disable`, `bundle_rollback`, `bundle_versions`. Operates on a TypeScript project in the agent's sandbox container. `bundle_build` runs `bun build` inside the container. `bundle_test` spawns a candidate bundle in a scratch loader isolate with a throwaway in-memory spine. `bundle_deploy` defaults to deploying *to the invoking agent's own bundle pointer* (self-editing is the default and is safe because of the static fallback). Optional `targetAgentId` for cross-agent deploys. Pre-deploy smoke test, deploy rate limiting, audit logging.
- **Sandbox container update**. The `SandboxContainer` Dockerfile vendors a snapshot of the bundle-authoring packages at a read-only mount path with integrity verification, so `bun install --ignore-scripts` and `bun build` work offline and resist supply-chain tampering by adversarial bundles.
- **Cache layering**. Worker Loader in-memory cache (hot path, keyed by content hash) → KV (authoritative bundle bytes) → R2 (sandbox scratch only, never on the runtime read path) → D1 registry (pointers and metadata, with `ctx.storage` cache of active version per agent). Content-addressed version IDs eliminate invalidation races. Bundles are stateless across turns, so isolate sharing between agents with identical content is safe — no warm state to leak.
- **Per-entry bundle version tagging**. Session entries record `bundleVersionId` (or `static`) of the brain that produced them, so session replay can reconstruct which brain authored each turn even after subsequent bundle changes. The DO stamps this tag, not the bundle, so it cannot be forged.
- **Out-of-band privileged endpoint** at `POST /bundle/disable` on bundle-enabled DOs (authenticated via existing `agent-auth`). Allows operators to clear an active bundle and force the static brain to run. Useful when `bundle_disable` from inside the agent cannot reach the agent (e.g., the bundle is stuck in a load loop).

## Capabilities

### New Capabilities

- `agent-bundle`: the bundle authoring API (`defineBundleAgent`), the `BundleEnv` type constraint, the bundle's small async-by-default runtime, the bundle default-export contract (handleTurn / handleClientEvent / handleAlarm / handleSessionCreated / metadata), and the secret-isolation invariants on the bundle side. Lives in `packages/agent-bundle`.
- `bundle-dispatch`: the `bundle` config field on `defineAgent`, the per-turn dispatch check inside the static `AgentDO` ("if active bundle exists, run it; else run static brain"), the brain-override semantics (bundle replaces static brain entirely for the duration of one turn), capability token minting per turn, version tagging of session entries, and the `SpineService` WorkerEntrypoint that bridges async bundle adapter clients (`SessionStoreClient`, `KvStoreClient`, `SchedulerClient`, `SessionChannel`) to the DO's existing sync `SessionStore`, `KvStore`, `Scheduler`, and `Transport`. Spine methods take a token first argument, derive identity from the verified payload, enforce per-turn RPC budgets, and provide send-only transport-out (incoming client events route through the bundle's `POST /client-event` endpoint, not through spine RPC).
- `llm-service`: host-side LLM provider proxy. Multi-provider routing, credential lookup from host env, token-first method signatures, per-agent inference rate limits, cost emission via spine RPC. Lives in `packages/agent-bundle`.
- `capability-service-pattern`: the structural convention for capability packages that hold secrets — `index.ts` (legacy static), `service.ts` (host WorkerEntrypoint with secrets), `client.ts` (bundle-side proxy), `schemas.ts` (shared tool schemas). Cost emission inside the service, attributed via verified token. Drift detection via schema content hash.
- `bundle-registry`: D1-backed version store, KV-backed content store, content-addressed version IDs, active-version pointer per agent, append-only deployment audit log, KV readback verification on deploy, D1 batch atomicity for multi-statement operations, self-seeding migration. Lives in `packages/bundle-registry`.
- `bundle-workshop`: agent-facing capability providing `bundle_init`, `bundle_build`, `bundle_test`, `bundle_deploy`, `bundle_disable`, `bundle_rollback`, `bundle_versions` tools. Default deploy target is the invoking agent's own bundle pointer (self-editing). Pre-deploy smoke test, deploy rate limiting, workshop tool audit logging. Sandbox container build flow with read-only vendored snapshot. Lives in `packages/bundle-workshop`.

### Modified Capabilities

- `agent-runtime-core`: gains an optional `bundle` config field on `AgentSetup` and a per-turn dispatch check inside the static `AgentDO` that delegates to a loaded bundle when one is registered. Additive — agents that omit the field exhibit no new behavior, no new code paths, no new dependencies. The static `SessionStore`, `Transport`, `AgentRuntime`, and `CapabilityHookContext` interfaces are unchanged. No async refactor of public surface.
- `define-agent-factory`: the `AgentSetup` shape gains an optional `bundle?: BundleConfig<TEnv>` field. Existing fields (`model`, `prompt`, `tools`, `capabilities`, etc.) are unchanged and still describe the agent's static brain (which serves as the always-available fallback). The factory still returns a DO class with the same constructor signature and registration shape.

## Impact

- **New packages**:
  - `packages/agent-bundle` (bundle authoring + bundle runtime + `LlmService` + `SpineService` WorkerEntrypoint + capability token utilities)
  - `packages/bundle-registry` (D1/KV registry, self-seeding migration)
  - `packages/bundle-workshop` (agent-facing authoring tools)
- **Modified packages**:
  - `packages/agent-runtime` — gains optional `bundle` config on `defineAgent`, gains the per-turn dispatch check, gains the SpineService bridge methods. **All additive. Static-only consumers see zero change.** No interface refactor, no async migration, no breaking changes.
  - `packages/tavily-web-search` — gains `service`, `client`, and `schemas` subpath exports while keeping its existing `index.ts` unchanged for static consumers. Pilot for the capability service pattern.
  - `packages/cloudflare-sandbox` — container Dockerfile updated to vendor workspace bundle-authoring packages at a read-only mount with integrity hash manifest.
- **No package modifications required for static-only consumers**: every existing CLAW capability package, every existing example, every existing static `defineAgent` consumer is unaffected. Today's `examples/basic-agent` works without changes. The async/sync boundary lives at the service binding edge (between bundle isolate and host DO), not inside the static runtime.
- **New wrangler bindings required for bundle-enabled agents only**: `worker_loaders` (LOADER), KV namespace for bundle bytes (BUNDLE_KV), D1 database for the registry (BUNDLE_DB), `AGENT_AUTH_KEY` secret (HMAC capability token key), and a service binding to `LlmService` plus any capability service entrypoints the bundle env exposes. Static-only agents declare none of these.
- **Secrets audit**: HMAC `AGENT_AUTH_KEY` and provider API keys must be reachable by their respective `WorkerEntrypoint` classes (host worker env) but never projected through `bundleEnv`. The type system enforces this at authoring time; the host worker's `bundleEnv` factory is the runtime gatekeeper. HKDF-derived per-service verify-only subkeys ensure that a single compromised service entrypoint cannot mint tokens for other services or for arbitrary identities.
- **Cost tracking**: bundle-side cost emission is attributed via the verified capability token at the spine, so cross-session cost forgery is structurally impossible. Static-agent cost emission is unchanged.
- **Container image rebuild cadence**: the sandbox container image must be rebuilt whenever the bundle-authoring package surface changes. CI gating.
- **New example**: `examples/bundle-agent` demonstrating a single agent with `bundle` config, the workshop, and the full edit → build → test → deploy → run loop including self-editing.
- **CLAUDE.md updates**: new sections covering the brain-override architecture, the bundle authoring story, the capability service pattern, the secrets-and-token security model, the four-layer cache hierarchy.
- **Out of scope for this change**: (a) GC / pruning of old KV bundle entries — deferred to a follow-up. (b) Cross-agent bundle discovery UX beyond the raw registry — deferred. (c) Non-Tavily capability service splits — only Tavily is piloted; other capability packages retain static-only surface for now. (d) HMAC key rotation — single secret with manual rotation via redeploy is acceptable for v1. (e) Multi-region clock-skew handling for capability token TTL — Cloudflare's NTP-synced edge clocks are bounded enough for the 5-minute TTL. (f) Generic loader-host DO that hosts many unrelated agents — that's a future architectural option built on top of this change, not required by it.
- **Supersedes**: the previous `add-agent-bundles` change. That change introduced a parallel SDK approach and accumulated three rounds of revisions trying to manage the resulting breaking changes; this change replaces it with the brain-override approach. The old change's specs and tasks should be archived once this change is approved. Most of the technical content (registry schema, capability token model, capability service pattern, workshop tools, KV readback) is reused; what changes is how the bundle integrates with the existing static runtime.
