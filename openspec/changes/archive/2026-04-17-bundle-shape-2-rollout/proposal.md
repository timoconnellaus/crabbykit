## Why

`tavily-web-search` is the only capability today that bundles can call. It got there via the "shape-2" 4-subpath split — `index` (legacy static), `service` (host `WorkerEntrypoint` holding credentials), `client` (bundle-side proxy reading `env.__BUNDLE_TOKEN`), `schemas` (shared TypeBox + drift-detection hash). Three other static-only capabilities — `file-tools`, `vector-memory`, `skills` — are stuck behind the bundle boundary because they have not been split. Bundles can declare them in `requiredCapabilities` (the catalog accepts the kebab-case ids), but with no `client` subpath there is nothing for the bundle to import, and with no `service` host-side wrapper there is no scope-checked verify path. Effectively, a bundle agent today can search the web but cannot read its own files, query its own memory, or load a skill — three capabilities that are central to how a real agent uses its environment.

There is a deeper problem the obvious "split each cap" framing hides. Today, when a bundle agent executes a tool, the tool runs **inside the bundle isolate**. The host's capability hook bus — `afterToolExecution` (observer) and `beforeInference` (mutator) — only fires on the **host's** tool pipeline. Capabilities like `vector-memory` (re-indexes `MEMORY.md` after `file_write`), `doom-loop-detection` (counts repeated tool calls), and `tool-output-truncation` (rewrites oversized tool results before they reach the model) all subscribe to those hooks. None of them fire for bundle-originated tool calls today. The static brain and the bundle brain were intended to be functionally equivalent — the bundle is just a different way to *run* the brain, not a different brain — but in their current state, every host capability whose behavior depends on observing or mutating tool execution silently degrades when the brain is moved into a bundle. Splitting `file-tools`, `vector-memory`, and `skills` without first solving the hook bridge would cement that gap as a "feature" of bundle agents — the static→bundle migration path would lose auto-reindexing, loop detection, output truncation, and skill conflict injection without warning.

The fix is to introduce a host-hook-bus bridge: the bundle runtime calls back into the host via spine after every tool execution and before every inference call, the host runs its full hook chain against the bundle's events, and the (possibly mutated) result feeds back into the bundle's loop. With the bridge in place, every existing host-side capability that subscribes to `afterToolExecution` or `beforeInference` continues to work for bundle agents without modification. The shape-2 split for the three target capabilities then becomes mechanical — the same pattern Tavily already proves out, with no per-cap workarounds for the hook gap because there is no gap. Functional parity between static and bundle is restored as a foundation, not as a deferred follow-up.

Three motivations make this the right next bundle-system change:

1. **Functional parity is non-negotiable.** Bundle ≡ static *functionally*. The bundle changes how the brain runs, not what it can do. Without the hook bridge, that invariant is silently broken for any agent using vector-memory, doom-loop-detection, tool-output-truncation, or skills — i.e., any non-trivial agent. Solving it first means the rest of the rollout doesn't ship a regression.
2. **Same work copied to three caps, plus one foundation.** With the bridge in place, the three capability splits are mechanical: lift tools into a `WorkerEntrypoint`, write a thin RPC client that reads `env.__BUNDLE_TOKEN`, share schemas via a `schemas.ts` exporting a `SCHEMA_CONTENT_HASH`, add `bundle-token` to `package.json`, point the consumer at `service`/`client` instead of the legacy static export. Tavily is a working reference. Three near-identical phases compress into one proposal far better than three separate proposals at ~30–40 minutes of authoring overhead each.
3. **Catalog and unified token already shipped.** `define-bundle-capability-catalog` validates `requiredCapabilities` at promotion and dispatch. `unify-bundle-capability-token` collapsed N tokens into one `__BUNDLE_TOKEN` whose `scope` array is derived from the validated catalog. A new shape-2 capability is now strictly *additive*: pick a kebab-case id, add it to your bundle's `requiredCapabilities`, verify with `requiredScope: "<id>"` in the service. The infrastructure that makes this cheap is already in place; this proposal is the first batch to consume it, atop the hook bridge.

The four phases land in order — the hook bridge first as Phase 0, then the three capability splits ordered by binding count (easiest first) so each phase de-risks the next.

- **Phase 0 — `bundle-host-hook-bridging`.** New spine methods `recordToolExecution(token, event)` and `processBeforeInference(token, messages)`. Bundle SDK runtime calls them in its tool execution wrapper and inference loop. Host runs its existing `afterToolExecutionHooks` and `beforeInferenceHooks` chains against the bundle events, returns possibly-mutated results. Static-brain code paths are unchanged.
- **Phase 1 — `skills`.** Single tool (`skill_load`). One D1 binding (skill registry) plus R2 read for skill content. No Workers AI, no Vectorize, no UI bridge.
- **Phase 2 — `vector-memory`.** Two tools (`memory_search`, `memory_get`). R2 read + Vectorize index + Workers AI embedder. Adds two native bindings beyond Phase 1. Auto-reindexing of `MEMORY.md`/`memory/*.md` works automatically because `vector-memory`'s existing `afterToolExecution` hook fires via the Phase 0 bridge.
- **Phase 3 — `file-tools`.** Nine tools (`file_*`). R2 read/write. UI mutation broadcast works automatically because the static capability's `afterToolExecution` hook fires via the Phase 0 bridge — no per-cap spine workaround needed.

Tavily missed `bundle-host` and `bundle-token` deps in its initial landing and had to add `bundle-token` retroactively in `unify-bundle-capability-token`. Each capability phase here adds them from the start.

## What Changes

### Phase 0 — host hook bus bridge

- **Two new SpineHost methods.** `agent-runtime/src/spine-host.ts`'s `SpineHost` interface gains `spineRecordToolExecution(caller, event): Promise<void>` and `spineProcessBeforeInference(caller, messages): Promise<AgentMessage[]>`. `AgentRuntime` (the DO that structurally satisfies `SpineHost`) implements them by delegating to its existing `afterToolExecutionHooks` and `beforeInferenceHooks` chains, wrapped through `withSpineBudget(caller, "<category>", fn)` for budget enforcement under fresh categories `"hook_after_tool"` and `"hook_before_inference"`.

- **Two new SpineService entrypoint methods.** `bundle-host/src/services/spine-service.ts` exposes `recordToolExecution(token, event)` and `processBeforeInference(token, messages)` as bundle-callable RPC methods. Both verify with `requiredScope: "spine"` (the bundle hook bridge is part of the spine surface, not a new scope), then delegate to the corresponding `SpineHost` method via the typed `DurableObjectStub<SpineHost>`. Errors sanitized through the existing `SpineService.sanitize` path.

- **Bundle SDK runtime calls the bridge in two places.** `bundle-sdk/src/runtime.ts`'s `runBundleTurn` (or its tool-execution wrapper, exact insertion point determined at implementation time):
  1. After every tool execution completes, calls `spine.recordToolExecution(token, { toolName, args, result, isError, durationMs })`. Fire-and-forget pattern (`waitUntil`-style or awaited; design.md picks). Failure of the bridge call SHALL NOT block the bundle's loop.
  2. Before every `AgentRuntime`/inference call, calls `messages = await spine.processBeforeInference(token, messages)` and uses the returned (possibly mutated) message array.

- **Static brain unchanged.** The host's existing inline `afterToolExecution` / `beforeInference` invocation in `AgentRuntime` runs identically for static-brain tool executions. The bridge methods are *additional* entry points into the same hook chains, not a replacement. A static-brain agent with `vector-memory` sees its hook fire from the host's tool pipeline; a bundle-brain agent with the same `vector-memory` capability sees the same hook fire from the bridge. Same hook code, same `CapabilityHookContext`, same observable behavior.

- **Hook context source-of-truth.** Bridge methods construct the `CapabilityHookContext` from the verified `caller` (aid + sid + nonce) and the host's current state — same as the inline static path. Hooks can't tell the difference between a host-pipeline event and a bridge-routed bundle event; they SHOULD NOT need to.

- **`ToolExecutionEvent` payload over the wire.** The bridge serializes the existing `ToolExecutionEvent` shape (`toolName`, `args`, `result`, `isError`, `durationMs`) as JSON. Tool args + results are already JSON-serializable per the static path's contract. No new payload schema invented; reuses what `afterToolExecution` already gets.

- **`AgentMessage[]` payload over the wire.** `processBeforeInference` accepts and returns `AgentMessage[]` (the existing pi-agent-core message type), serialized as JSON over RPC. Bundle's inference loop swaps in the returned array verbatim.

- **Budget enforcement.** Both new spine methods enter the per-turn budget tracker under fresh categories. Default category caps are conservative defaults that won't fire for normal usage; a misbehaving bundle that floods bridge calls hits the cap and fails fast rather than degrading host performance.

### Phase 1–3 — three capability shape-2 splits

- **`@crabbykit/skills` adds three subpaths.** `src/service.ts` exports a `SkillsService` `WorkerEntrypoint` with one method (`load(token, args, schemaHash) -> { content: string }`) that verifies `requiredScope: "skills"`, looks up the installed-skill record, fetches content from R2, strips frontmatter, and returns the body. `src/client.ts` exports a `skillsClient(options: { service: Service<SkillsService> })` factory returning a bundle-side `Capability` with a `skill_load` tool that proxies via RPC. `src/schemas.ts` exports the tool name, description, args schema, and `SCHEMA_CONTENT_HASH = "skills-schemas-v1"`. The legacy `src/index.ts` (static `skills(...)` factory) keeps all lifecycle hooks (`onConnect`, `beforeInference`, `afterToolExecution`, `configNamespaces`, `httpHandlers`) — only the `skill_load` tool is dropped from its `tools` return when the consumer wires the bundle client instead. `package.json` exports map gains `./service`, `./client`, `./schemas`. Dependencies add `@crabbykit/bundle-token: workspace:*`. **The static capability's `beforeInference` (skill conflict injection) and `afterToolExecution` (dirty tracking) fire automatically for bundle-originated events via the Phase 0 bridge** — no per-cap workaround.

- **`@crabbykit/vector-memory` adds three subpaths.** `src/service.ts` exports a `VectorMemoryService` `WorkerEntrypoint` with two methods: `search(token, args, schemaHash) -> { results: ... }` (verifies `requiredScope: "vector-memory"`, embeds query via Workers AI, queries Vectorize, fetches matched chunks from R2) and `get(token, args, schemaHash) -> { content: string }` (verifies, reads R2 file, returns content with byte-cap). `src/client.ts` exports `vectorMemoryClient(options: { service: Service<VectorMemoryService> })` returning a `Capability` with `memory_search` and `memory_get` tools plus the existing `promptSections` (the prompt is content-only, safe to live in the client). `src/schemas.ts` exports tool names, descriptions, args schemas, `SCHEMA_CONTENT_HASH = "vector-memory-schemas-v1"`. The static `src/index.ts` keeps the `afterToolExecution` indexing hook host-side. Dependencies add `@crabbykit/bundle-token: workspace:*`. The Workers AI binding (`Ai`), Vectorize binding (`VectorizeIndex`), and R2 bucket all stay on the service env — the bundle client never touches them. **Auto-reindexing of `MEMORY.md`/`memory/*.md` after bundle-originated `file_write`/`file_edit`/`file_delete` works automatically via the Phase 0 bridge.**

- **`@crabbykit/file-tools` adds three subpaths.** `src/service.ts` exports a `FileToolsService` `WorkerEntrypoint` with nine methods, one per tool (`read`, `write`, `edit`, `delete`, `copy`, `move`, `list`, `tree`, `find`), each verifying `requiredScope: "file-tools"`. `src/client.ts` exports `fileToolsClient(options: { service: Service<FileToolsService> })` returning a `Capability` whose `tools` proxy via RPC. `src/schemas.ts` exports tool names, descriptions, args schemas (one per tool), and `SCHEMA_CONTENT_HASH = "file-tools-schemas-v1"`. The static `src/index.ts` keeps `onAction` (UI bridge — host-only by nature, no bundle equivalent needed) and `afterToolExecution` (mutation broadcast). Dependencies add `@crabbykit/bundle-token: workspace:*`. **UI mutation broadcast for bundle-originated writes works automatically via the Phase 0 bridge** — the static capability's `broadcastAgentMutation` hook fires against the bundle's tool event and emits `file_changed` to the UI, no per-method spine call needed inside the service.

- **Catalog ids and scopes.** `skills`, `vector-memory`, and `file-tools` are the three new kebab-case scope strings. They match the existing static `Capability.id` for each capability and the bundle `requiredCapabilities[].id` declaration. `unify-bundle-capability-token` already injects `requiredCapabilities[].id` into the minted `scope` array verbatim — no token-side changes needed. A bundle that declares `[{ id: "skills" }]` gets `scope: ["spine", "llm", "skills"]`; the `SkillsService.verify` check passes; the `FileToolsService.verify` check on the same token would fail with `ERR_SCOPE_DENIED`. The dispatcher's existing `validateRequiredCapabilities` check already errors at promotion time if a bundle declares a capability id the host doesn't know about — the host's `knownCapabilityIds` set must be widened to include `skills`, `vector-memory`, `file-tools` in each phase's `examples/basic-agent` wiring (and any consumer's wiring). This is one line per consumer per phase and lives outside the package boundary.

- **Schema drift hash per capability.** Each `schemas.ts` exports a `SCHEMA_CONTENT_HASH` constant. The bundle-side client passes the hash on every RPC call; the service compares against its own constant and throws `ERR_SCHEMA_VERSION` on mismatch. Same defensive check Tavily uses. Hashes are version-string-style (`"skills-schemas-v1"`), bumped manually when the args schemas change in a way that would silently mistype old bundles against a new host.

- **Cost emission via spine.** None of the three capabilities have paid bindings today. No cost emission to wire. If a future phase introduces a paid binding the service emits via `spine.emitCost(token, ...)` exactly as Tavily does.

- **Consumer wiring per phase.** `examples/basic-agent` gains a `bundleCapabilities` block in its `defineAgent` for the phase being landed, mirroring the existing Tavily wiring. Each phase's tasks include the consumer-side wiring so the basic-agent example demonstrates the new bundle capability end-to-end. Static-brain wiring of the same example continues to work unchanged because the legacy `index.ts` factory is preserved.

- **No mode-awareness in this proposal.** Bundle dispatch is not mode-aware in v1. The `bundle-runtime-surface` follow-up proposal adds mode-aware dispatch for bundles. Within this proposal, all three capabilities behave identically inside vs outside a mode.

## Capabilities

### New Capabilities

- `bundle-host-hook-bridging`: spine-mediated bridge so the host's existing `afterToolExecution` (observer) and `beforeInference` (mutator) hook chains fire against bundle-originated tool execution and inference events. Foundation for Phases 1–3 and for any future shape-2 capability split. Establishes that bundle and static brains are *functionally* equivalent — the bundle changes the runtime, not the surface area of capability behavior.
- `bundle-shape-2-skills`: bundle-side `skills` capability (the `skill_load` tool, RPC-proxied to host `SkillsService`). Establishes the catalog-id-to-scope binding for skills, the schema-drift hash protocol, and that lifecycle hooks (sync, conflict injection, dirty-tracking, configNamespaces, httpHandlers) stay host-side and fire via the hook bridge for bundle events.
- `bundle-shape-2-vector-memory`: bundle-side `vector-memory` capability (`memory_search`, `memory_get` tools, RPC-proxied to host `VectorMemoryService`). Establishes that `promptSections` can live on the bundle client when it is content-only, and that auto-reindexing on file mutations works for bundle agents via the hook bridge.
- `bundle-shape-2-file-tools`: bundle-side `file-tools` capability (nine `file_*` tools, RPC-proxied to host `FileToolsService`). Establishes that the static capability's `afterToolExecution` mutation-broadcast hook fires for bundle events via the bridge — no per-method spine call inside the service.

### Modified Capabilities

- `agent-bundles`: the dispatcher's `knownCapabilityIds` set widens (per consumer wiring) to include `skills`, `vector-memory`, `file-tools` in each landing phase. No code changes inside `bundle-host` or `agent-runtime`'s dispatch path — `validateRequiredCapabilities` already accepts an arbitrary kebab-case id set passed in by the consumer. The change is a documentation-and-example update on the dispatch side. Spine-related modifications (the two new bridge methods) are scoped to the `bundle-host-hook-bridging` new capability spec, not folded into `agent-bundles`.

## Impact

- **Modified packages**:
  - **Phase 0**: `packages/runtime/agent-runtime/` (SpineHost interface widens with two methods; `AgentRuntime` implements them by delegating to existing hook arrays). `packages/runtime/bundle-host/` (`SpineService` exposes the two new bundle-callable RPC methods). `packages/runtime/bundle-sdk/` (runtime tool-execution wrapper + inference loop call the bridge methods).
  - **Phase 1**: `packages/capabilities/skills/` adds `src/service.ts`, `src/client.ts`, `src/schemas.ts`; updates `package.json` exports + dependencies.
  - **Phase 2**: `packages/capabilities/vector-memory/` adds the same three files and updates `package.json`. Service holds `Ai`, `VectorizeIndex`, R2 bucket.
  - **Phase 3**: `packages/capabilities/file-tools/` adds the same three files and updates `package.json`. Service holds R2 bucket only — UI broadcast goes through the static capability's hook firing via the bridge.
  - `examples/basic-agent/` per phase: wires the new service binding in `wrangler.toml`, registers the service via the worker entry, swaps the consumer-side capability registration for the bundle-client variant when the bundle brain is the active path. Static-brain consumers can keep using the legacy factory.
- **Unchanged packages**: `packages/runtime/bundle-token/`, `packages/runtime/bundle-registry/`, `packages/runtime/agent-workshop/`. Token primitives, registry storage, and workshop tools are untouched.
- **Consumer wiring (Phases 1–3)**: `wrangler.toml` gains a service binding per phase per consumer (one of `SKILLS_SERVICE`, `VECTOR_MEMORY_SERVICE`, `FILE_TOOLS_SERVICE`). Worker entry exports the corresponding `WorkerEntrypoint` class. `defineAgent` wires the service into the bundle-side capability via the existing options-object pattern Tavily uses. Static-brain consumers see no wiring changes.
- **Wire-format**: per-RPC for capability calls, each carries `(token, args, schemaHash)`. For bridge calls: `recordToolExecution(token, event)` carries `ToolExecutionEvent` JSON; `processBeforeInference(token, messages)` carries `AgentMessage[]` JSON in and out. No new payload fields on the unified token.
- **Security posture**: per-capability scope check via `verifyToken(..., { requiredScope: "<id>" })`. Bridge methods verify under `"spine"` (they're part of the spine surface). No new attack surface.
- **Bundle env namespace**: zero growth. All capabilities and the bridge use the existing `__BUNDLE_TOKEN`.
- **Hot-path cost**:
  - Bridge `recordToolExecution`: one extra RPC per tool execution. Roughly the cost of a single capability call. Pure addition over the static pipeline. For tool-heavy agents this matters; the budget tracker enforces a cap.
  - Bridge `processBeforeInference`: one extra RPC per inference call. Inference is already RPC-heavy (LLM call typically dwarfs the bridge call), so the relative overhead is small.
  - Per-tool RPC for capability calls: one verify (HMAC + nonce + scope) plus one schema-hash compare. Same as Tavily.
- **Cold-start cost**: each service does one `deriveVerifyOnlySubkey` on first call, cached on the entrypoint instance. Same pattern as Tavily.
- **Out of scope**:
  - Mode-aware dispatch for bundle tools. Belongs in `bundle-runtime-surface`.
  - Lifting `onConnect`, `httpHandlers`, `configNamespaces`, `onAction`, `onSessionCreated`, `onTurnEnd`, `onAgentEnd`, `onConfigChange`, `onScheduleFire` into bundle-aware paths. These hooks are either host-pipeline-only (lifecycle events specific to the host's connection or session) or expose host-only surfaces (HTTP, config UI). The bridge specifically targets the two hooks whose semantics are tied to tool-execution and inference — the only two that affect bundle/static functional equivalence for the capabilities in scope.
  - Cost emission for capabilities that have no paid binding today (`skills`, `vector-memory`, `file-tools`).
  - Compatibility shims for consumers swapping static for bundle. Per CLAUDE.md, no compat layer.
  - A typed `ctx.capabilities.<id>.<method>(...)` bundle-side surface as an alternative to tool calls.
- **Risk profile**: medium for Phase 0 (new runtime path, hot-path cost, mutator semantics); low-medium per shape-2 phase. Phase 0 is the foundation — its design correctness affects the other three. design.md enumerates the open questions on bridge semantics (await vs fire-and-forget for observer hooks, latency budget for mutator hooks, error propagation, ordering vs parallelism within the hook chain).
- **Phase atomicity**: each phase is one atomic commit per CLAW conventions, lands independently after Phase 0. Phase 0 must land first; Phases 1–3 can land in any order after that, though the proposal recommends 1→2→3 for binding-count-based de-risking. If Phase 2 turns sideways, Phases 0 and 1 stay shipped.
