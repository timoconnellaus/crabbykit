## Why

The `agent-bundle` package currently contains two halves with opposite constraints sharing a single package:

- **`src/host/`** â€” the host-side dispatcher. Runs on the worker that owns the agent DO. Holds the HKDF master secret, binds native `WorkerLoader`, `DurableObjectNamespace`, and service bindings, and mints per-turn capability tokens. Imports `cloudflare:workers`. Must never be loaded inside a bundle isolate.
- **`src/bundle/`** â€” the bundle authoring SDK. Runs *inside* a Worker Loader isolate. Accepts only a serializable `BundleEnv`. Must never see native bindings (structured clone across the isolate boundary fails on `Ai`, `R2Bucket`, etc.). The SDK is what bundle authors install and compile with `bun build --target=browser`.

These are not two layers of one concept. They are two programs with different runtimes, different dependencies, different installation footprints, and different security postures. Packaging them together creates real problems:

1. **Bundle authors install host code they never use.** A bundle project that imports `@claw-for-cloudflare/agent-bundle` for `defineBundleAgent` pulls in the entire `src/host/` tree as transitive source, including the SpineService WorkerEntrypoint class that references `cloudflare:workers`. Tree-shaking compiles this away for the final bundle output, but type-checking and local tooling still load the host code. Bundle project `tsconfig.json` complains about types it can't use.
2. **The type system cannot prevent host/SDK confusion.** A bundle author can accidentally import a SpineService utility from the main barrel and it "works" at authoring time but fails inside the isolate. The package boundary is currently imaginary â€” enforced only by convention and a few `__tests__` comments.
3. **Security posture leaks.** The host half holds the HKDF master secret and mints tokens. The bundle half verifies tokens but can only derive per-service subkeys from the *verified* payload, not forge new ones. Shipping both halves as one package muddles the trust boundary â€” any reviewer asking "where is the master secret used?" has to read both halves of the same package to answer.
4. **Install footprint.** A bundle project wants the smallest possible dev dependency for bundle authoring. `@claw-for-cloudflare/agent-bundle` today includes the dispatcher, token minting, budget tracker, in-memory registry (test helper), SpineService, LlmService, capability service pattern utilities â€” most of which are irrelevant to authoring.

This change splits `agent-bundle` into two separate packages â€” `bundle-sdk` (authoring) and `bundle-host` (dispatcher) â€” and extracts the `SpineHost` interface (the contract the host DO implements to receive bridged calls from SpineService) into `agent-runtime` where all runtime contracts live. The split is a pure refactor. No behavior changes. No API changes visible to consumers other than the package name they import from.

This proposal depends on `reorganize-packages-by-role` having landed. The new packages are placed in `packages/runtime/bundle-host/` and `packages/runtime/bundle-sdk/` â€” the runtime bucket created by the prior proposal. If this proposal lands first, the packages go in the flat `packages/` layout and must be moved when the reorg lands. Prefer the reorg first.

## What Changes

- **Split `@claw-for-cloudflare/agent-bundle` into two packages:**
  - **`@claw-for-cloudflare/bundle-sdk`** â€” contains the current `src/bundle/` contents: `defineBundleAgent`, `BundleContext`, `BundleEnv`, `BundleCapability`, `BundleExport`, all bundle-authoring types, the bundle's small async runtime (`buildBundleContext`, `runBundleTurn`), the bundle prompt builder, the bundle spine client classes. Also inherits the `build.ts` script and the generated `dist/bundle-runtime-source.js` artifact exporting `BUNDLE_RUNTIME_SOURCE` and `BUNDLE_RUNTIME_HASH` constants â€” these are the built-string form of the SDK runtime that gets injected into every user bundle at workshop build time. Runtime dependencies: `@claw-for-cloudflare/bundle-token`, `@claw-for-cloudflare/agent-runtime` (type-only for re-exports), `@claw-for-cloudflare/agent-core` (pi-agent-core fork for the LLM loop inside the isolate), `@claw-for-cloudflare/ai` (pi-ai providers for model access), `@sinclair/typebox`. Zero `cloudflare:workers` runtime imports; `@cloudflare/workers-types` only as a devDependency for `Service<T>` type references. This is the package bundle authors install.
  - **`@claw-for-cloudflare/bundle-host`** â€” contains the current `src/host/` contents: `BundleDispatcher`, `SpineService`, `LlmService`, `BudgetTracker`, `BundleConfig` type (the full consumer-facing one with `registry`, `loader`, `autoRebuild` fields), `InMemoryBundleRegistry` test helper, `bundle-builder.ts` (the auto-rebuild helper that uses `@cloudflare/worker-bundler` to rebuild a bundle when the injected runtime-source hash drifts from the host's current hash), and the host-side capability token minting utilities. Runtime dependencies: `@claw-for-cloudflare/bundle-token`, `@claw-for-cloudflare/bundle-sdk` (type-only for `BundleExport` contract reference and for the `./runtime-source` subpath carrying `BUNDLE_RUNTIME_SOURCE` + `BUNDLE_RUNTIME_HASH` constants), `@claw-for-cloudflare/agent-runtime`, `@claw-for-cloudflare/bundle-registry`, `@cloudflare/worker-bundler`. Imports `cloudflare:workers` (for `WorkerEntrypoint`, `DurableObjectNamespace`, `WorkerLoader` types), holds the HKDF master-secret handling, and is only consumed by the worker that hosts agent DOs. This is the package worker authors install when they want bundle brain override on their agents.
- **Security utilities (`src/security/capability-token.ts`) move to `bundle-host`.** The token minting (`mintToken`, `deriveSubkey`) is host-only. The token verification (`verifyToken`, `NonceTracker`) is used by both halves â€” host SpineService verifies tokens, bundle SpineClient structure validates received responses. The verify-only surface is re-exported from `bundle-sdk` so bundles can validate without pulling in mint code. Tree-shaking cannot be relied on to remove mint from a bundle build because the mint and verify functions share a module. Explicit split: `mint.ts` stays in `bundle-host`, `verify.ts` is in a small shared module that both packages depend on â€” most naturally a third micro-package `bundle-token-core`, OR a subpath export from `bundle-host` that `bundle-sdk` depends on only at type level. (See design.md for the decision between these options.)
- **Extract `SpineHost` interface into `@claw-for-cloudflare/agent-runtime`.** The `SpineHost` interface currently lives in `packages/agent-bundle/src/host/spine-service.ts` and describes the methods the host DO must implement to receive bridged SpineService calls (`spineAppendEntry`, `spineKvGet`, `spineBroadcast`, `spineEmitCost`, etc.). This is a runtime contract â€” the DO implements it, the SpineService calls it. It belongs in `agent-runtime` alongside `SqlStore`, `KvStore`, `Scheduler`, `Transport` interfaces. Move the type, update `bundle-host` to import it from `@claw-for-cloudflare/agent-runtime`, update `agent-do.ts` in the runtime to structurally satisfy the interface (the methods already exist; this is a type-level cleanup).
- **`packages/agent-runtime/src/bundle-config.ts` stays in `agent-runtime`, unchanged.** This file was added in commit `a70d70b` to break a circular dependency: `agent-runtime` needs minimal types (`BundleVersionMetadata`, `BundleVersionInfo`, `CreateBundleVersionOpts`) for the drift-detection path in `AgentDO.initBundleDispatch`, and `agent-bundle` depends on `agent-runtime`, so `agent-runtime` cannot reach back for the fuller types. Post-split, the same circular constraint applies (`bundle-host` depends on `agent-runtime`), so these minimal types remain inlined in `agent-runtime/src/bundle-config.ts`. The fuller `BundleConfig<TEnv>` (with `registry`, `loader`, `autoRebuild` fields) lives in `bundle-host/src/bundle-config.ts` and imports the minimal types from `@claw-for-cloudflare/agent-runtime` to keep them structurally aligned. This split is deliberate and documented in the `agent-runtime-core` spec as a circular-dep-break.
- **Bundle dispatcher entry-point moves to `bundle-host/src/dispatcher.ts`.** The current `packages/agent-bundle/src/host/bundle-dispatcher.ts` becomes the primary export of `@claw-for-cloudflare/bundle-host`. It retains its signature â€” accepts a `BundleConfig` object and returns a dispatch function that the DO invokes per turn.
- **The `bundle` config field on `defineAgent` stays where it is**, but its type is now sourced from `@claw-for-cloudflare/bundle-host` instead of `@claw-for-cloudflare/agent-bundle`. Agents that use bundle brain override change one import:
  ```ts
  // before
  import { type BundleConfig } from "@claw-for-cloudflare/agent-bundle";
  // after
  import { type BundleConfig } from "@claw-for-cloudflare/bundle-host";
  ```
  Bundle authoring imports change analogously:
  ```ts
  // before
  import { defineBundleAgent, type BundleContext } from "@claw-for-cloudflare/agent-bundle";
  // after
  import { defineBundleAgent, type BundleContext } from "@claw-for-cloudflare/bundle-sdk";
  ```
- **`@claw-for-cloudflare/agent-bundle` is deleted.** No stub, no re-export package, no alias. The repo is greenfield and does not maintain legacy compatibility shims. The old name is removed from the workspace, the old directory is removed from `packages/runtime/`. All internal consumers update their imports in the same change.
- **Update `@claw-for-cloudflare/agent-workshop` to consume the split packages.** The workshop imports `BundleConfig` and dispatcher utilities from `bundle-host`, and imports bundle authoring types (for the bundle test harness) from `bundle-sdk`. Two imports where there was previously one. Note: commit `a70d70b` already extracted the bundle build helpers (`loadBundleFiles`/`buildBundle`/`encodeEnvelope`) from workshop into `agent-bundle/src/host/bundle-builder.ts`, so the workshop is already thinner than it was when this proposal was drafted. Workshop tools now delegate bundle building to `bundle-builder.ts` rather than owning the logic themselves.
- **Update `packages/capabilities/tavily-web-search/src/client.ts`** to import bundle-side types from `@claw-for-cloudflare/bundle-sdk` instead of `@claw-for-cloudflare/agent-bundle`. This is the pilot capability for the capability service pattern and is the first demonstrator of "bundle-side client code that depends on the SDK only."
- **Update `packages/runtime/bundle-registry`** â€” it references `BundleMetadata` and related types. Audit the imports; they either come from `bundle-sdk` (if they're authoring-side) or stay in `bundle-registry` itself (if they're registry-internal). Expect most to move to `bundle-sdk`.
- **Update `examples/basic-agent`** if it references `@claw-for-cloudflare/agent-bundle` directly. Likely it doesn't â€” it uses the `defineAgent` `bundle:` field via `agent-runtime` re-exports â€” but verify.
- **Update tests.** Every test that imports from `@claw-for-cloudflare/agent-bundle` updates to the split packages. Tests are spread across `packages/agent-bundle/src/**/__tests__/` (including the newly-added `bundle-builder.test.ts` and `decode-bundle-payload.test.ts`), `packages/agent-runtime/test/integration/bundle-dispatch.test.ts`, `packages/agent-runtime/src/test-helpers/fake-worker-loader.ts`, `packages/agent-runtime/test/fixtures/bundle-sources.ts`, `packages/agent-runtime/test/helpers/bundle-client.ts`, and `packages/agent-workshop`. Test files themselves may move between the new packages depending on which half they exercise â€” `bundle-builder.test.ts` and `decode-bundle-payload.test.ts` move to `bundle-host`; test helpers in `agent-runtime` stay where they are but update imports to reference the split packages.

## Capabilities

### New Capabilities

- **`bundle-sdk`**: the bundle authoring API (`defineBundleAgent`), the `BundleEnv` constraint type, the bundle's small async runtime, the bundle default-export contract, the bundle prompt builder, and the bundle-side spine client classes. Packaged independently of the host dispatcher so bundle authors install only what they need. Lives in `packages/runtime/bundle-sdk/`.
- **`bundle-host`**: the host-side dispatcher, `SpineService` and `LlmService` WorkerEntrypoint classes, `BudgetTracker`, per-turn capability token minting via HKDF, and the `BundleConfig` type consumed by `defineAgent`'s `bundle:` field. Imports `cloudflare:workers`. Holds the HKDF master secret via env. Lives in `packages/runtime/bundle-host/`.

### Modified Capabilities

- **`agent-bundles`** (the existing bundle-brain-override capability): its constituent implementation moves from a single package to two packages. All scenarios previously specified against `@claw-for-cloudflare/agent-bundle` are re-homed to the correct sub-package. The behavior, the protocol, the security model, and the scenarios themselves are unchanged. Spec deltas rewrite package references and add one new requirement: "The bundle SDK and bundle host dispatcher SHALL be separately installable packages with distinct import paths."
- **`agent-runtime-core`**: gains the `SpineHost` interface as a runtime contract alongside `SqlStore`, `KvStore`, `Scheduler`, `Transport`. No behavior change; the type was already implicitly required by `bundle-host`'s consumption of `AgentDO` â€” this change makes the contract explicit and imports cleaner.

### Removed Capabilities

None. The `agent-bundle` package is deleted as a package but its constituent capabilities (`agent-bundle`, `bundle-dispatch`, `llm-service`, `capability-service-pattern`) are all retained and re-homed to `bundle-sdk` or `bundle-host` as appropriate.

## Impact

- **New packages**:
  - `packages/runtime/bundle-token/` â€” verify-only token primitives micro-package (see design.md)
  - `packages/runtime/bundle-sdk/` â€” bundle authoring SDK, including the `build.ts` script that compiles the SDK runtime into `dist/bundle-runtime-source.js` and the `./runtime-source` subpath export carrying `BUNDLE_RUNTIME_SOURCE` and `BUNDLE_RUNTIME_HASH` constants. Runtime deps include pi-agent-core and pi-ai forks for the in-isolate LLM loop.
  - `packages/runtime/bundle-host/` â€” host-side dispatcher, services, `BundleConfig` full type, auto-rebuild helper (`bundle-builder.ts`) using `@cloudflare/worker-bundler`, mint security utilities
- **Deleted packages**:
  - `packages/runtime/agent-bundle/` â€” split into the three above, directory removed entirely
- **Modified packages**:
  - `packages/runtime/agent-runtime/` â€” gains the `SpineHost` interface export. Additive. The existing `src/bundle-config.ts` (circular-dep-break types for `BundleVersionMetadata`/`BundleVersionInfo`/`CreateBundleVersionOpts`) remains in place â€” this is not affected by the split and is deliberately kept where it is.
  - `packages/runtime/agent-workshop/` â€” imports split across `bundle-host` and `bundle-sdk`. Workshop tools already delegate bundle building to `bundle-builder.ts` post-commit-`a70d70b`, so the refactor is smaller than originally scoped.
  - `packages/runtime/bundle-registry/` â€” import audit, likely unchanged semantically. Note that `BundleRegistry` interface gained optional `getVersion` + `createVersion` methods in commit `a70d70b`; the post-split registry type continues to live in `bundle-registry` and both `bundle-host` and `bundle-sdk` consume it from there.
  - `packages/capabilities/tavily-web-search/` â€” bundle-side client updates one import.
  - `examples/basic-agent/` â€” import audit. Its `src/worker.ts` was modified in commit `0ee3c4d` and may already import specific subpaths; the split must preserve or update those.
  - `e2e/agent-runtime/` â€” import audit, update any references.
- **Installation footprint changes**:
  - A consumer that only ships static agents with no bundle brain override installs nothing new and removes no existing dependency â€” they never depended on `@claw-for-cloudflare/agent-bundle` in the first place.
  - A consumer that uses bundle brain override on the host worker installs `@claw-for-cloudflare/bundle-host` (which pulls in `@claw-for-cloudflare/bundle-token`, `@claw-for-cloudflare/bundle-registry`, `@claw-for-cloudflare/bundle-sdk` type-only for contract references, and `@cloudflare/worker-bundler`) instead of `@claw-for-cloudflare/agent-bundle`. The total transitive footprint is similar â€” the savings are in reviewer legibility and trust boundary clarity, not bytes.
  - A bundle author installs `@claw-for-cloudflare/bundle-sdk` instead of `@claw-for-cloudflare/agent-bundle`. Because bundles run a full LLM loop inside the isolate, `bundle-sdk` must ship `@claw-for-cloudflare/agent-core` (pi-agent-core fork) and `@claw-for-cloudflare/ai` (pi-ai fork) as runtime deps. It is not "tiny" â€” it is roughly the same runtime size as the old authoring surface of `agent-bundle/bundle`. The win is what is NOT in the install tree: `bundle-host`'s dispatcher, mint utilities, `@cloudflare/worker-bundler`, and WorkerEntrypoint classes that the bundle author has no business seeing.
- **No runtime behavior changes**: every existing test scenario for `add-bundle-brain-override` continues to produce identical outputs. The split is purely structural.
- **Security posture unchanged**: the HKDF master secret still lives only in `bundle-host`'s env. Token minting still happens only in `bundle-host`. Token verification still happens in `SpineService` (host-side) and in LlmService. The bundle SDK does not gain any ability to mint tokens â€” the mint code is not exported from or importable by `bundle-sdk`.
- **Token verify-only surface**: both halves need the ability to verify a token (the host verifies incoming bundle RPCs; the bundle validates incoming spine responses if any such validation is added in future). To avoid both halves depending on `bundle-host` (which would defeat the split), the verify utilities live in a small shared location. Two options evaluated in `design.md`:
  1. **Subpath export from `bundle-host`**: `bundle-sdk` depends on `bundle-host` but only imports types + verify helpers via `@claw-for-cloudflare/bundle-host/verify`. Bundle code that tree-shakes properly will drop the mint side. Downside: `bundle-sdk` transitively includes `bundle-host` in its install graph, weakening the split's install-footprint benefit.
  2. **Third micro-package `bundle-token`**: contains token types, HKDF subkey derivation, verification helpers, nonce tracking. Both `bundle-host` and `bundle-sdk` depend on it. Mint code stays only in `bundle-host`, which re-exports token types. This is cleaner and preserves the "bundle author installs minimal tree" property.
  - **Decision: option 2 (third micro-package).** Adds one package but preserves the install-footprint win and the trust boundary clarity. See design.md for rationale.
- **Documentation updates**:
  - `CLAUDE.md` architecture rules section describing the bundle system updates to reference the split packages instead of the single `agent-bundle` package.
  - `README.md` packages table updates.
  - Any architectural diagrams or docs in the `add-bundle-brain-override` change's artifacts are NOT modified â€” that change is archived, its artifacts are historical record.
- **Test updates**: integration tests in `packages/runtime/agent-runtime/test/integration/bundle-dispatch.test.ts` update imports; unit tests in the old `agent-bundle/src/**/__tests__/` are split between the two new packages according to which half they exercise.
- **Out of scope**:
  - Renaming `BundleContext` or `SpineService` or any other internal type. This is a mechanical split, not a redesign.
  - Generalizing the capability service pattern to additional capabilities beyond Tavily. That is the existing roadmap item "shape-2 rollout" and is not coupled to this split.
  - Adding a new bundle capability token model (unified vs per-service tokens). Still an open question; this proposal does not touch it.
  - Mode-aware bundle dispatch. Still v1.1 as noted in `add-bundle-brain-override`.
- **Depends on**: `reorganize-packages-by-role` having landed, so the `packages/runtime/` bucket exists as a target.
