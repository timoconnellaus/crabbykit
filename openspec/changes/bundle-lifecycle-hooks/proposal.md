## Why

Bundles can author **inference** hooks (`beforeInference`, `beforeToolExecution`, `afterToolExecution`) and **alarm/session/client-event** hooks via the bundle-runtime-surface bridges. They CANNOT author the **per-turn-completion**, **per-connect**, or **disposal** hooks — `Capability.afterTurn`, `Capability.hooks.onConnect`, `Capability.dispose` — nor the AgentDO-level `onTurnEnd` / `onAgentEnd` setup hooks. This is the dominant blocker for porting channel capabilities (`channel-telegram` delivers the final assistant message via `afterTurn` — without it, the channel can't be authored as a bundle) and for any audit/cost-reporting/analytics capability that needs to observe terminal turn state.

Concrete consequence: `channel-telegram` currently can't ship as a bundle because the bot's "send final message back to chat" step lives in `afterTurn`. A bundle author building a Discord-channel-as-bundle, an audit-log capability, or a Sentry-integration capability hits the same wall — three separate "but how do I observe the end of a turn?" requests.

This change closes that gap. Bundles declare the same per-capability `afterTurn` / `hooks.onConnect` / `dispose` hooks that static capabilities declare, plus the setup-level `onTurnEnd` / `onAgentEnd` parity hooks; the host bridges each one to the bundle isolate over the established host→bundle fetch dispatch (same envelope shape used by `/alarm`, `/session-created`, `/client-event`); existing static plumbing fires unchanged for static capabilities.

## What Changes

- Add `BundleCapability.afterTurn?: (ctx: BundleAfterTurnContext, sessionId: string, finalText: string) => Promise<void>` matching the static top-level field. Host fires AFTER the existing static-capability `dispatchAfterTurn` walk (`agent-runtime.ts:3064-3117`), inside the same `waitUntil`-tracked `Promise` so dispatch failure is bounded by the existing `pendingAsyncOps` lifecycle.
- Add `BundleCapability.hooks.onConnect?: (ctx: BundleOnConnectContext) => Promise<void>` matching the static `hooks.onConnect`. Host fires from each of the four `fireOnConnectHooks` call sites (`agent-runtime.ts:1629` initial-connect, `1638` hibernation-restore, `1817` session-switch, `1843` new-session). Each bundle capability's handler runs in the bundle isolate.
- Add `BundleCapability.dispose?: () => Promise<void>` matching the static top-level field. Host fires once per `disposeCapabilities` call (`agent-runtime.ts:2802` — invoked from `1722` last-WebSocket-close and `2658` agent_end). The dispose dispatch envelope SHALL carry NO `sessionId` because `disposeCapabilities` is per-DO not per-session; the bundle SDK iterates `setup.capabilities(env)` and invokes each declared `dispose()`.
- Add `BundleAgentSetup.onTurnEnd?: (messages: AgentMessage[], toolResults: BundleToolResult[]) => void | Promise<void>` matching the static `AgentDelegate.onTurnEnd` (`agent-runtime.ts:2985`). Host serializes `event.toolResults` host-side via a new `projectToolResultsForBundle` helper into a JSON-clonable `BundleToolResult[]` projection; non-projectable shapes are dropped from individual entries (NOT silently dropped wholesale — each entry's projection failure logs `[BundleDispatch] kind: "lifecycle_on_turn_end" outcome: "tool_result_projection_failed"`).
- Add `BundleAgentSetup.onAgentEnd?: (messages: AgentMessage[]) => void | Promise<void>` matching the static `AgentDelegate.onAgentEnd` (`agent-runtime.ts:3001`).
- **Five new bundle endpoints** — `/after-turn`, `/on-connect`, `/dispose`, `/on-turn-end`, `/on-agent-end`. Each accepts the standard JSON envelope `{sessionId?, agentId, ...payload}` and returns 204 on success or `{error: string}` on bundle-side throw. Same dispatch shape as `/alarm` / `/session-created` / `/client-event`.
- **Five new optional callback hooks on `AgentRuntimeOptions`** — `bundleAfterTurnHandler?`, `bundleOnConnectHandler?`, `bundleDisposeHandler?`, `bundleOnTurnEndHandler?`, `bundleOnAgentEndHandler?`. Mirror the existing `bundleClientEventHandler` / `bundleAlarmHandler` / `bundleSessionCreatedHandler` pattern (`agent-runtime.ts:448-472`). Each is populated in `agent-do.ts` (alongside the existing `1886-1917` block), which has access to `BundleDispatcher` state and `composeWorkerLoaderConfig`. **`agent-runtime.ts` does NOT import `bundle-host`** — the architecture rule `runtime/ → runtime/` is preserved.
- **Extend `BundleMetadata.lifecycleHooks`** with five new boolean flags — `onTurnEnd?: boolean`, `onAgentEnd?: boolean`, `afterTurn?: boolean`, `onConnect?: boolean`, `dispose?: boolean`. Host (via `agent-do.ts`'s callback wiring) reads each flag at dispatch time; if false, skips Worker Loader instantiation entirely (same pattern as bundle-runtime-surface). `afterTurn` / `onConnect` / `dispose` are aggregates — true if ANY capability in the bundle's `setup.capabilities(probeEnv)` walk declares the hook. `onTurnEnd` / `onAgentEnd` are setup-level — true if `setup.onTurnEnd` / `setup.onAgentEnd` is defined.
- **Mode-filter interaction is a known cost.** Build-time aggregation does NOT account for the dispatch-time mode filter (Decision 14 of bundle-runtime-surface — bundle-side filtering happens INSIDE the isolate). A bundle whose only `afterTurn`-bearing capability is filtered out by the active mode still incurs the dispatch cost; the bundle SDK iterates and finds nothing to invoke. Documented as honest cost rather than a bug; "zero overhead" applies only to bundles that didn't declare the hook AT BUILD TIME.
- **Expose `BundleDispatcher.getActiveLifecycleFlags()`** returning the active bundle's `BundleMetadata.lifecycleHooks` snapshot (or `null` when no bundle is active). The callback wiring in `agent-do.ts` consults this getter before invoking `composeWorkerLoaderConfig`.
- **`composeWorkerLoaderConfig` SHALL be the shared dispatch helper** for the five new endpoints. Pre-existing drift: `BundleDispatcher.dispatchClientEvent` (`bundle-host/src/dispatcher.ts:340-381`) hand-rolls the loader config inline. **Prerequisite cleanup task lands in this proposal** — refactor `dispatchClientEvent` to call `composeWorkerLoaderConfig` first, then add the five new dispatch sites that reuse it. After this proposal lands, the helper is the single source of truth for all loader configs (matched-set lint rule added in task 5.3).
- **Token mint extension for session-less dispatch.** `mintToken` (`bundle-host/src/token.ts`) currently requires `sessionId: string`; this proposal extends the payload with `sid: string | null`. Spine methods that need a session call a new `requireSession(caller)` helper at the top of the method body that throws `ERR_SESSION_REQUIRED` (new code in the existing `SpineErrorCode` union) when `caller.sid === null`. Used by `/dispose` exclusively in v1; future session-less endpoints (e.g. agent-level config sync) reuse the same shape.
- **No new spine budget categories.** The proposal initially considered per-endpoint budget categories but `BudgetCategory` is a closed string-literal union (`bundle-host/src/budget-tracker.ts:51-59`) keyed on operation TYPE (`sql`, `kv`, `broadcast`, `inspection`, `hook_after_tool`, `hook_before_inference`, `hook_before_tool`), not caller endpoint. Lifecycle handlers calling `spine.appendEntry` flow through the existing `sql` budget; calling `channel.broadcast` flows through `broadcast`. The existing per-turn caps already bound runaway lifecycle handlers via the same mechanism that bounds runaway turn-loop handlers.
- **Per-handler timeout** via `BundleConfig.lifecycleHookTimeoutMs?: number` (default 5 000 ms — matches the `configHookTimeoutMs` precedent for non-turn UX-bound dispatches). Lives in `packages/runtime/agent-runtime/src/bundle-config.ts` (where `BundleConfig` is defined). The 30 000 ms `httpDispatchTimeoutMs` is separately scoped to HTTP-route dispatch and does NOT apply here. Timeout-on-expire logs `outcome: "timeout"` and the host event proceeds; Worker Loader fetch is not cancelled (Workers runtime constraint).
- **`afterTurn` parameter shape.** Host concatenates the final assistant message text via the existing `extractFinalAssistantText` helper (imported at `agent-runtime.ts:3076` from `agent-runtime-helpers.ts`) and forwards it as `finalText: string`. Empty string when the turn terminated without assistant text.
- **Context shapes — fresh types, NOT extending `BundleHookContext`.** `BundleAfterTurnContext`, `BundleOnConnectContext`, `BundleDisposeContext`, `BundleTurnEndContext`, `BundleAgentEndContext` are all NEW types defined in `bundle-sdk/src/types.ts`. None extend `BundleHookContext` because that type carries `hookBridge` (a turn-loop concept that would generate phantom events if invoked outside a turn — the same reason `BundleSpineClientLifecycle` excludes it per bundle-runtime-surface Decision 8). Each new context exposes only what makes sense for its endpoint: per-cap contexts carry `agentId, sessionId?, capabilityId, kvStore, channel?, spine: BundleSpineClientLifecycle, publicUrl, emitCost, agentConfig?`; setup-level contexts carry `agentId, sessionId?, spine, publicUrl`.
- **Documented v1 parity gaps** (NOT exposed on the new contexts, mirroring `BundleHttpContext` precedent): static `AgentContext` carries `schedules`, `rateLimit`, `requestFromClient`, `broadcastToAll`, `notifyBundlePointerChanged`. The new bundle-side contexts don't expose these — bundle authors needing them have alternatives: schedules via `setup.onAlarm` + `ctx.scheduler` from `BundleContext`; rateLimit via host-side wrapper; broadcasts via `ctx.channel.broadcast` (session) or `ctx.channel.broadcastGlobal()`; pointer notification via the host (bundle-internal pointer changes go through `setActive` host-side). Documented honest gaps, not silent breaks.
- **`dispose` is connectionless and dispatched-after-static-kickoff.** Host's `disposeCapabilities` (`agent-runtime.ts:2802-2811`) is **synchronous-returning** — static `dispose()` calls are kicked off with `.catch()` attached but NOT awaited. The bundle `/dispose` dispatch SHALL be queued as a separate `runtimeContext.waitUntil(...)` registration AND added to `pendingAsyncOps` (mirroring `dispatchAfterTurn` at `agent-runtime.ts:3046-3048`) so the DO does not become eligible for hibernation until the bundle dispatch resolves. The "static walks first" claim is qualified: static disposes are KICKED OFF first; bundle dispose dispatch runs concurrently with in-flight static disposes. This matches static behavior for siblings (per-capability disposes also race each other).
- **`afterTurn` lifetime is the existing `dispatchAfterTurn` `waitUntil` registration.** The bundle dispatch is awaited INSIDE `dispatchAfterTurn` (after the static loop completes), so the existing `dispatchPromise` registered with `waitUntil` (`agent-runtime.ts:3046-3048`) covers both static and bundle execution. No new `waitUntil` registration needed.
- **`onConnect` lifetime.** `fireOnConnectHooks` is invoked with `.catch(...)` at four call sites without `await` (e.g. `agent-runtime.ts:1629`). The bundle `/on-connect` dispatch SHALL be queued via `runtimeContext.waitUntil(...)` AND `pendingAsyncOps` registration in the same shape as `dispose`. The static walk's per-cap awaits inside `fireOnConnectHooks` itself (line 2814+) complete before the bundle dispatch is queued — so `await staticWalk; queue bundleDispatch` ordering holds within the function, but the function itself is non-awaited at its call sites.
- **`onTurnEnd` and `onAgentEnd` setup-level dispatch.** Static delegates fire from `handleAgentEvent` at lines 2985 / 3001 with their result's `.catch(...)` attached but NOT awaited. The bundle dispatch SHALL be queued via the same `waitUntil` + `pendingAsyncOps` pattern. Ordering: the static delegate's promise is registered first (kicked off), then the bundle dispatch is queued. Both run concurrently; the host event loop proceeds without blocking on either.
- **Bundle handler errors never block host event handling.** All five dispatchers wrap the loader fetch in `catch` blocks that log structured `[BundleDispatch] kind: "lifecycle_*" outcome: "error"` and resolve to undefined. The dispatchPromise registered with `waitUntil` resolves cleanly even on bundle failure.
- **No retry on lifecycle dispatch failure.** All five hook dispatches are fire-and-await with one shot — failures are logged but do NOT count toward `consecutiveFailures` (same disposition as `/alarm` / `/session-created` / `/client-event`). These are observation-only paths; retrying could fire side effects twice.
- **`afterTurn` ordering within the bundle isolate.** `setup.capabilities(env)` registration order determines invocation order. Static and bundle `afterTurn` chains are independent; the host always kicks off the static chain first inside `dispatchAfterTurn`, awaits it, then awaits the bundle dispatch.
- **Telemetry.** Structured `[BundleDispatch]` logs with `kind: "lifecycle_after_turn" | "lifecycle_on_connect" | "lifecycle_dispose" | "lifecycle_on_turn_end" | "lifecycle_on_agent_end"` discriminators (matches the established prefix-with-kind pattern from prior proposals). Each carries `agentId`, `sessionId?` (omitted for `dispose` and `on-agent-end`), `bundleVersionId`, and `outcome: "ok" | "timeout" | "error" | "tool_result_projection_failed"`.
- **No reserved-id concerns.** These hooks don't introduce identifiers — they extend behavior on existing capability declarations. No collision validators required. `lifecycleHooks` field shape is internal metadata that bundle authors don't write directly.

## Capabilities

### New Capabilities

- `bundle-lifecycle-hooks`: bundle-side authoring API for `BundleCapability.afterTurn`, `BundleCapability.hooks.onConnect`, `BundleCapability.dispose`, plus `BundleAgentSetup.onTurnEnd` and `BundleAgentSetup.onAgentEnd`. Host bridges each over the established `/lifecycle-*` fetch dispatch envelope, reusing `composeWorkerLoaderConfig` (after a prerequisite refactor of `dispatchClientEvent`). Lifecycle declarations are detected at build time and projected into `BundleMetadata.lifecycleHooks`; dispatch is skipped entirely when the bundle didn't declare the hook (zero overhead for legacy bundles, plus a documented edge case for mode-filtered capabilities). Five fresh context types — `BundleAfterTurnContext`, `BundleOnConnectContext`, `BundleDisposeContext`, `BundleTurnEndContext`, `BundleAgentEndContext` — replace the temptation to reuse `BundleHookContext`, which would leak the turn-loop `hookBridge` into non-turn dispatches. New `mintToken` shape supports `sid: null` for session-less `/dispose` dispatch; new `ERR_SESSION_REQUIRED` guard at the top of session-scoped spine methods makes the constraint loud rather than silent.

### Modified Capabilities

_None._ Pure addition. The `bundle-runtime-surface` lifecycle-hooks substrate (`BundleMetadata.lifecycleHooks`, the `composeWorkerLoaderConfig` envelope helper, the `[BundleDispatch]` telemetry prefix) is reused without modification beyond the prerequisite cleanup of `dispatchClientEvent`.

### NOT in scope (deferred)

- **`channel-telegram` (or any channel) port to a bundle.** This proposal unblocks the port; the actual port is a separate workstream once `bundle-mcp-servers` and `bundle-capability-storage` are also addressed.
- **Channel-as-bundle abstraction.** A bundle declaring itself as a channel (with `defineChannel`-equivalent ergonomics) is a distinct surface — covered by a future `bundle-channels` proposal.
- **`onScheduleFire` parity with static.** Already shipped via `setup.onAlarm` in bundle-runtime-surface phase 2.
- **Bundle declaring its own setup-level `validateAuth`.** Auth happens before any dispatch — bundles cannot intercept it without compromising the auth boundary.
- **`beforeToolExecution` / `afterToolExecution` / `beforeInference` parity.** Already covered by bundle-runtime-surface (host bridge + bundle-side handlers).
- **Bundle-side per-endpoint spine budget categories.** Existing per-type budget (`sql`, `kv`, `broadcast`) bounds runaway lifecycle handlers via the same mechanism that bounds runaway turn-loop handlers.
- **Mode-filter integration with build-time aggregation.** Documented as known cost — bundle declared `afterTurn` but mode filters out the declaring capability still incurs dispatch cost. Future optimization could thread the active mode into the build-time aggregation, but defer.
