## Why

CLAW agents today are statically defined at worker compile time: the system prompt, tool set, capability list, and model config are all passed to `defineAgent()` when the worker is built, and changing any of them requires redeploying the worker. This blocks three things we want to support: (1) agents that can author and iterate on other agents (and eventually themselves), (2) per-deployment agent variants that don't require per-variant workers, and (3) a credible self-extending agent story where the LLM authors new behavior at runtime. The building blocks exist — Cloudflare Worker Loader can host JS on demand, the `AgentRuntime` three-layer architecture already separates platform-agnostic logic from CF-specific plumbing via adapter interfaces, and a POC in `examples/loader-agent` has demonstrated that a thin DurableObject can hot-reload a "brain" from R2. The missing piece is a first-class pathway: a bundle format, a loader-backed agent entrypoint, a rigorous secrets-and-authorization story, a real adapter refactor so `AgentRuntime` can run inside a loader isolate, and a sandbox-based workshop that lets agents edit, build, test, and deploy bundles without leaving their own session.

## What Changes

- **New consumer API**: `defineLoaderAgent<Env>({ loader, registry, bundleEnv, fallback, authKey, hostCapabilities? })` alongside the existing `defineAgent()`. Returns a DO class that loads its runtime from a registry-backed bundle at per-turn dispatch time instead of from statically-imported TS modules. Existing `defineAgent()` consumers are unaffected.
- **New bundle format**: `defineAgentBundle<BundleEnv>({ model, prompt, tools, capabilities, metadata? })` — authored in TypeScript, compiled to a single JS file via `bun build`, loaded into a Worker Loader isolate at runtime. Same declarative shape as `defineAgent()` minus DO-specific fields. `BundleEnv` is type-restricted to prevent bundles from accessing raw native bindings (`Ai`, `R2Bucket`, `DurableObjectNamespace`, `WorkerLoader`, `VectorizeIndex`, `D1Database`) that can't cross isolate boundaries or shouldn't hold secrets. Lives in `packages/agent-runtime/src/bundle/` and is exported via the new `@crabbykit/agent-runtime/bundle` subpath.
- **Spine/loader split in `agent-runtime`**: `AgentRuntime` moves into the loader bundle, constructed with RPC-backed adapters that call a new `SpineService` WorkerEntrypoint exposed by the host DO. The DO owns persistence, alarms, and client-facing WebSocket transport; tools, capabilities, prompt sections, and the LLM loop all run inside the loader isolate. **This requires a real adapter-layer refactor** (see Phase 0.5 below), not just new RPC wrappers — `SessionStore`'s surface must become async to cross JSRPC, and `Transport`'s client→bundle message routing must be redesigned because its current callback-registration interface (`onMessage`, `onClose`, `onOpen`) can't survive a service binding boundary.
- **Per-turn capability token authorization**: every `SpineService`, `LlmService`, and capability-service RPC method takes a sealed HMAC token as its first argument. The host worker mints the token when dispatching a turn into the loader, binding `{agentId, sessionId, expiresAt, nonce}` to that specific invocation. Services verify the token on every call and derive `agentId`/`sessionId` from the verified payload; caller-supplied session IDs in other arguments are ignored. This makes cross-session/cross-agent forgery structurally impossible — a bundle cannot mint a valid token without the HMAC key, which lives only in the host worker's env.
- **Host-side `LlmService`**: a WorkerEntrypoint that proxies inference calls for OpenRouter, Anthropic, OpenAI, and Workers AI. Holds provider API keys in its own env. Bundles declare `model: { provider, modelId }` without apiKey and the runtime routes through `env.LLM_SERVICE` via service binding, passing a capability token on every call. Lives in `packages/agent-bundle` alongside `defineLoaderAgent` so host workers pick it up with a single dependency.
- **Capability service pattern**: secret-holding capabilities split into three subpath exports — `index.ts` (legacy, unchanged), `service.ts` (WorkerEntrypoint with credentials), `client.ts` (bundle-side capability factory with statically-declared tool schemas that RPC into the service). Pilot: `packages/tavily-web-search`. Cost emission moves into the service, attributed via the verified capability token, so bundles can neither suppress nor fabricate cost events.
- **Bundle registry**: D1-backed version store with content-addressed bundle IDs (SHA-256 of the compiled artifact), a mutable per-agent active-version pointer, and an append-only deployment audit log. Compiled bundle bytes live in a KV namespace keyed by version ID; R2 is used only as the agent's sandbox workspace, never on the runtime read path. **Deploys use read-back verification**: `bundle_deploy` writes to KV, polls `kv.get()` until the bytes are visible, *then* updates the D1 pointer — handling KV's documented cross-location eventual consistency window. Self-seeding migration following the `skill-registry` pattern.
- **Bundle workshop capability**: `packages/bundle-workshop` gives agents six new tools — `bundle_init`, `bundle_build`, `bundle_test`, `bundle_deploy`, `bundle_rollback`, `bundle_versions` — that operate on a TypeScript project in the agent's sandbox container. `bundle_build` runs `bun build` inside the container. `bundle_test` spawns a subagent from the just-built artifact via Worker Loader with a scratch spine. `bundle_deploy` hashes the artifact, runs a pre-deploy smoke test, verifies KV readback, registers the version in D1, and updates the active pointer. Depends on existing `sandbox` and `subagent` packages.
- **Container image update**: the `SandboxContainer` Dockerfile vendors a snapshot of the `@crabbykit/*` bundle-authoring packages at a read-only mount path so `bun install` inside the container can resolve workspace imports via `file:` references. Vendored snapshot is mounted read-only and package integrity hashes are verified before `bun build` runs, closing the supply-chain attack surface. Rebuilds on SDK release.
- **Cache layering**: four layers — Worker Loader in-memory cache (hot path, keyed by content hash) → KV (authoritative runtime store) → R2 (sandbox scratch, not read during turns) → D1 (version metadata and active pointer). Content-addressed version IDs eliminate invalidation races.
- **Safety rails** (Option A protection, Option B readiness): out-of-band privileged factory-reset HTTP endpoint on every loader-backed DO; per-entry bundle version tagging so session replay is reconstructable across deploys; deploy rate-limiting per agent to bound runaway deploy loops; structured audit log entry for every workshop tool invocation. These protect subagent targets under Option A (a bricked subagent still needs human-recoverable recovery) and are forward-compatible with Option B self-editing.
- **First deployment model (Option A)**: the first ship treats the workshop as a capability on an *existing* static agent that produces *subagent* bundles, not a self-editing runtime. A static parent agent gains the tools to author and run child agents of arbitrary shape. The self-editing switch (parent agent's bundle path points at its own active version) is a configuration flag layered on top in a later change, gated by the factory-reset endpoint introduced here.
- **New example**: `examples/bundle-workshop-agent` demonstrating the full edit → build → test → deploy loop and showing the parent/subagent architecture.

## Capabilities

### New Capabilities

- `agent-bundle`: the bundle format, `defineAgentBundle()` API, `BundleEnv` type-level restrictions, secret-isolation invariants, and the contract the bundle runtime honors (what a bundle's default export must look like for the loader host to invoke it).
- `loader-agent`: the `defineLoaderAgent()` consumer API and its DO runtime. Covers per-turn bundle loading, Worker Loader cache key derivation from version IDs, spine RPC dispatch, fallback bundle semantics for cold starts and registry misses, the host-side env → bundle-side env projection, active-version pointer cached in DO storage, per-entry version tagging, and the out-of-band factory-reset endpoint.
- `spine-service`: the `SpineService` WorkerEntrypoint that exposes SQL store, KV store, scheduler, transport-out, and cost-emission RPC surfaces for bundle-side `AgentRuntime` adapters. Every method takes a verified capability token; identity is derived from the token, not from caller arguments. Covers the contract between the DO (persistence owner) and the loader-loaded runtime.
- `llm-service`: host-side LLM provider proxy. Multi-provider routing (OpenRouter, Anthropic, OpenAI, Workers AI), credential lookup from host env, invocation surface bundles call via `env.LLM_SERVICE` service binding with a capability token. Replaces the direct pi-ai call path for loader-backed agents.
- `capability-service-pattern`: the structural convention for capability packages that hold secrets. Split into `index.ts` (static agents, legacy), `service.ts` (WorkerEntrypoint holding secrets, cost-emitting, token-verifying), `client.ts` (bundle-side thin proxy with static tool schemas). Covers the shared-schema contract between service and client so the bundle knows tool shapes at compile time without seeing the implementation.
- `bundle-registry`: D1-backed version store, KV-backed content store, content-addressed version IDs, active-version pointer per agent, append-only deployment audit log, self-seeding migration, KV readback verification on deploy. Consumed by `loader-agent` for runtime lookup and by `bundle-workshop` for deploy operations.
- `bundle-workshop`: agent-facing capability providing `bundle_init`, `bundle_build`, `bundle_test`, `bundle_deploy`, `bundle_rollback`, `bundle_versions` tools plus deploy rate-limiting and per-tool audit logging. Runs `bun build` inside the sandbox container. Spawns test subagents from scratch loader isolates. Depends on `sandbox`, `subagent`, `bundle-registry`. Does NOT hold or expose bundle-side secrets — it reads built artifacts from the container filesystem and writes to registry + KV.

### Modified Capabilities

- `agent-runtime-core`: the adapter layer is refactored as a prerequisite for the loader-backed path. `SessionStore` becomes async across its entire surface (current synchronous `SqlStore.exec()`/`SqlResult.toArray()` iterator shape does not survive JSRPC); `Transport`'s incoming-message routing moves entirely to the DO with a new bundle-side `handleClientMessage` entry; isolate lifecycle and per-session warm-state semantics are pinned down. Static agents continue to work unchanged after the refactor, but their call sites must be audited and awaited where they previously relied on sync `SessionStore` access.

## Impact

- **New packages**: `packages/agent-bundle` (contains `defineLoaderAgent`, `LlmService`, and the host-side loader dispatch machinery), `packages/bundle-registry` (D1/KV registry), `packages/bundle-workshop` (agent-facing authoring tools). `SpineService` and `defineAgentBundle` live inside `packages/agent-runtime` under new subpath exports rather than in separate packages — this keeps the bundle-authoring side on one package root and the host side on one package root.
- **Modified packages**: `packages/agent-runtime` gains `/bundle` and `/spine` subpath exports, plus the adapter async refactor in Phase 0.5. `packages/tavily-web-search` gains `/service` and `/client` subpath exports while keeping its existing `index.ts` unchanged for static consumers. `packages/cloudflare-sandbox` container Dockerfile updated to mount a read-only vendored workspace package snapshot with integrity verification. `packages/sandbox` tools unchanged but used by `bundle-workshop`. `packages/subagent` unchanged — `bundle_test` uses a scratch loader invocation directly rather than adding a loader-backed subagent profile (Open Question 5 resolution).
- **New wrangler bindings required**: host workers that adopt `defineLoaderAgent` must configure `worker_loaders`, a KV namespace for bundle bytes, a D1 database for the registry, and an `AGENT_AUTH_KEY` secret (HMAC key for capability tokens). These are additive — existing static agents are unaffected.
- **Secrets audit**: the bundle security model requires that no provider API keys, capability credentials, or HMAC keys are passed through `bundleEnv`. The type system enforces this at authoring time but the host worker's `bundleEnv` factory is the ultimate gatekeeper. Documentation must make this contract explicit. The HMAC key in particular must be reachable by the host DO (to mint tokens) and by `SpineService`/`LlmService`/capability services (to verify them), but NEVER by the bundle env projection.
- **Cost tracking changes**: cost emission for loader-backed agents moves from inside-the-bundle to inside-the-service, with session attribution derived from the verified capability token (not caller-supplied). Static agents' cost emission path is unchanged. Both paths land in the same session store via the same `emitCost` surface.
- **Container image rebuild cadence**: the sandbox container image must be rebuilt whenever the `@crabbykit/*` bundle-authoring packages change their public surface. Release process implications.
- **Example coverage**: a new example (`examples/bundle-workshop-agent`) demonstrating the Option A model. The existing `examples/loader-agent` POC is retained as a minimal reference and is not folded in.
- **CLAUDE.md updates**: the "What the SDK Provides Today" section, the "Project Structure" section, and the "Architecture Rules" section all need new entries covering bundle authoring, the spine/loader split, the secrets isolation contract, the capability token authorization model, and the capability service pattern.
- **Out of scope for this change**: (a) the self-editing switch that points a parent agent's bundle path at its own active version — deferred to a follow-up change, but the factory-reset endpoint this change adds is the prerequisite it unblocks. (b) Bundle GC / pruning of old KV entries — deferred. (c) Cross-agent bundle discovery UX beyond the raw registry — deferred. (d) Non-Tavily capability service splits — only Tavily is piloted; other capability packages retain their static-only surface for now. (e) HMAC key rotation — a single `AGENT_AUTH_KEY` secret suffices for v1; rotation is follow-up work.
